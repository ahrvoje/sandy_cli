TOML Configuration Reference

Complete settings for AppContainer and Restricted Token modes — every key, its default, and whether it is mandatory, optional, or unavailable.

Configuration Fundamentals

Every config must have a [sandbox] section. Everything else defaults to deny-all / disabled. Omitting a setting never grants more access. Wrong-mode keys, unknown keys, and unknown sections are all configuration errors (exit 128).

📐 Minimal valid config

[sandbox]
token = 'appcontainer'

Two lines. All other sections are optional. No access is granted by omission — defaults are deny-all / disabled.

⚠️ Config limits

Config filemax 1 MB
Path lengthmax 32,768 chars
Rules per sectionmax 256 in [allow], [deny], [registry]
⛔ Wrong-mode keys are rejected

Using an AC-only key (e.g. network, system_dirs) in an RT config, or an RT-only key (named_pipes, integrity) in an AC config, is a configuration error → exit 128. Unknown keys/sections are also rejected.

[allow] — File and folder grants

both modes

All keys optional (default []). All grants are recursive except peek (directory itself only, non-recursive).

KeyDefaultACRTPermission granted
peek[]OptionalOptionalList directory + stat only — non-recursive, directory itself only
read[]OptionalOptionalRead files, list dirs (recursive). Does NOT grant execute.
write[]OptionalOptionalCreate/modify files (recursive). Does NOT grant read.
execute[]OptionalOptionalRead + execute files, list dirs (recursive)
append[]OptionalOptionalAppend-only. Cannot overwrite or read.
delete[]OptionalOptionalDelete only. No read or write.
all[]OptionalOptionalFull access: read+write+execute+delete. Strips WRITE_DAC, WRITE_OWNER, FILE_DELETE_CHILD.
💡 Allow-inside-deny pipeline

You can carve exempt subtrees inside denied areas. All allow/deny entries are merged and sorted by depth — deny applied before allow at each level. An allow inside a deny automatically strips the deny ACEs first. Example: deny.all = ['C:\repos'] + allow.all = ['C:\repos\snipps'].

[deny] — Block specific permissions

both modes

All keys optional (default []). peek is not in [deny]. Deny is recursive and surgical — only the specified permission type is blocked. deny.write does NOT block delete.

KeyDefaultAC mechanismRT mechanism
read[]Mask reduction (ALLOW bits subtracted)Real DENY_ACCESS ACE
write[]Mask reductionReal DENY_ACCESS ACE. Does NOT block delete.
execute[]Mask reductionReal DENY_ACCESS ACE
append[]Mask reductionReal DENY_ACCESS ACE
delete[]Mask reductionReal DENY_ACCESS ACE
all[]Mask reduction (all bits)Real DENY_ACCESS ACE (all bits)
⚠️ AppContainer deny is mask reduction, not DENY ACEs

The Windows kernel ignores DENY_ACCESS ACEs for AppContainer SIDs. Sandy implements AC deny by subtracting bits from the existing ALLOW mask and setting PROTECTED_DACL to block re-inheritance from parent grants.

Restricted only

Not available in AppContainer — AC uses a fixed private registry hive automatically. Most HKLM\Software and HKCU keys are already readable via BUILTIN\Users in RT.

🔒 Limits are fail-closed

If a limit is configured but cannot be enforced (job object failure), Sandy terminates the child and exits with code 129. The sandbox never runs with unenforced limits.
